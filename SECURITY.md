# Security Policy

## Security Model

SecretKeyRandomizer follows an open source security model. Source code,
algorithms, presets, encoding choices, entropy calculations, UI behavior, and
build commands may be public. The generated secret value is the data that must
remain private.

Secrets generated with sufficient entropy using a cryptographically secure
random source are computationally infeasible to brute-force in practice.

## Threat Model

An attacker may know:

- The source code
- The Python version
- The libraries
- The presets
- The encoding algorithms
- The entropy calculation logic
- The UI behavior
- The build method

An attacker must not know the actual secret value generated by the user's local
CSPRNG at generation time.

## Randomness Source

The core generator uses Python standard library secure randomness APIs:

- `secrets.token_bytes`
- `secrets.choice`
- `uuid.uuid4` only for UUID-v4 output

The core generator does not use Python's `random` module.

## Entropy Policy

Byte-based output uses:

```text
entropy_bits = byte_length * 8
```

Custom charset output uses:

```text
entropy_bits = length * log2(charset_size)
```

Prefix and suffix values are not counted because they are normally public and
predictable.

Default security presets target at least 256 bits of entropy. Salt defaults to
128 bits because salts do not need to be secret, but should still be random.

## No Persistence Policy

By default, SecretKeyRandomizer does not:

- Store generated secrets
- Keep secret history
- Write secrets to a database or config file
- Log secrets
- Send telemetry
- Perform analytics
- Use cloud sync
- Make network requests

## Clipboard Warning

Clipboard is not secure storage. Copied secrets can be read by other
applications. The app only copies when the user manually clicks Copy, and it can
clear the clipboard after 30 seconds.

## What This Tool Helps Protect Against

- Weak secret generation
- Low entropy defaults
- Predictable token generation
- Insecure use of non-CSPRNG randomness
- Wrong default byte lengths for common secret types
- Unsafe real-secret examples in public repositories

## What This Tool Does Not Protect Against

- Malware or keyloggers on the user's computer
- Clipboard-monitoring malware
- Screen capture malware
- Accidental commits of generated secrets
- Unsafe storage in `.env` or deployment systems
- Secrets written to logs by other applications
- Sharing secrets with other people
- A broken or compromised operating system randomness source

## Reporting Vulnerabilities

If you find a vulnerability, open a private security advisory or contact the
maintainer through the repository's preferred security channel. Do not include
real generated secrets in reports.
